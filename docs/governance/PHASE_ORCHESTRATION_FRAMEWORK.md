# HAVDM – PHASE ORCHESTRATION FRAMEWORK (HARD MODE++)

This document defines the mandatory governance process for initiating any new Phase in HAVDM.

It is:

- Phase-agnostic
- Version-aware
- Architecture-aware
- Beta and stable compatible
- Solo-developer optimized
- AI-constrained
- Risk-governed

It must be used before initiating any new Phase branch.

# GOVERNANCE RELATIONSHIP WITH ai_rules.md (MANDATORY)

This framework governs:

- Phase planning
- Slice design
- Packaging discipline
- Refactor scoping
- Version bump validation
- Stop conditions
- Medium Gate enforcement
- Release readiness

`ai_rules.md` governs:

- Implementation safety
- Immutable state updates
- React stability rules
- DSL blast-radius contracts
- Test execution & reporting workflow
- Git feature script enforcement
- HACS alignment rules
- Document storage standards

Precedence boundaries:

- Implementation safety rules in `ai_rules.md` cannot be overridden.
- This framework controls scope, sequencing, and packaging.
- If a conflict arises, implementation safety wins.
- All slice prompts generated by this framework MUST explicitly reference compliance with `ai_rules.md`.

Never guess version progression.

## 2) Assumption Register

- Assumption
- Rationale
- Confidence level
- Validation path in repo

---

## 3) No Code Changes

Planning + audit only.

---

## 4) Refactor Governance

Refactors allowed only if:

- Required for the phase
- OR materially reduce regression/performance risk
- AND slice-sized + test-protected

No broad rewrites.

If refactor requires DSL changes, DSL Change Contract from `ai_rules.md` applies.

---

## 5) Solo Operator Clarity Rule

Every slice and commit must include:

- What to do
- Where to do it
- What to run
- What success looks like
- What to do if it fails

If uncertainty arises → STOP and ask.
